## 🏗️ Архитектура и Логика работы

Проект представляет собой голосового помощника на базе модульной сети:

### 1. **Клиент: ESP32 (Прошивка Xiaozhi)**
- **Роль:** Передача звука со встроенного микрофона на сервер и воспроизведение ответов через динамики.
- **Ключевые состояния:**
  - `Idle` (Standby): Устройство слушает Wake Word («Компьютер»). 
  - `Listening`: Горячая точка - сжатие звука в потоковый OPUS и отправка по WebSocket на `.102`.
  - `Speaking`: Проигрывание входящего потока OPUS с сервера.
- **Настройка звука (`ES8311`):** Зафиксирована на 80% громкости для предотвращения аппаратных сбоев усилителя при пиковых нагрузках питания.

### 2. **Шлюз: Nanobot-Gateway (Узел `.102` — 🐳 Docker)**
- **Среда:** Работает внутри Docker-контейнера **`nanobot-gateway`**.
- **Ollama:** На этом узле запущен отдельный локальный Docker-контейнер **`ollama`** (через который идет запрос в Cloud).
- **Директория:** `/home/alexander/nanobot`
- **Роль:** Оркестрация событий. Сердце обработки — `nanobot/nanobot/channels/voice_server.py`.
- **Логика стриминга:**
  1. WebSocket считывает PCM кадры от ESP32 (через декодер).
  2. Замеряет уход в тишину (RMS < 50) для автоматической отсечки (`Auto-stopping`).
  3. Отсылает аудио на узел `.111` для перевода в текст.
  4. Полученный текст кормит LLM (**Ollama Cloud**).
  5. Текст ответа проговаривается в обратный звуковой API и льется на ESP32.

### 3. **Служба STT: Speaches (Узел `.111` — 🐳 Docker)**
- **Среда:** Работает в Docker-контейнере **`speaches`**.
- **Директория:** `/mnt/vm_hdd/docker-compose/speaches`
- **Роль:** Распознавание речи на базе Whisper (`large-v3`).
- **GPU Ускорение:** Работает на видеокарте P104-100.
- **Особенности:** Добавлен `WHISPER__COMPUTE_TYPE=int8_float16`, что даёт оптимальную скорость инференса без урезания качества на чипах Pascal.

## 🔧 Критичные заметки по отладке

1. **Аппаратный дедлок (Тишина / rms=9-10):** Возникает, если прошивка пытается сыграть локальный звук (например `OGG_POPUP` / «бимп») **в ту же миллисекунду**, что запускается запись звука с микрофона. I2S шиши глушат кодек, и он шлёт нули. Все переходы в и из `Listening` нужно делать без параллельных звуковых маркеров.
2. **Перегрузка по току:** Крайне опасно ставить громкость на 100% на слабом питании платы — чип защиты глушит микродинамики. 80-85% — золотой стандарт.
3. **Медленные ответы:** 55 секунд генерирования ответа Ollama обычно вызваны очередями в облаке Ollama Cloud (превью режим), генерированием сверхдлинного списка или тем, что gateway накапливает буфер ответа целиком до конца.

4. **Проблема однократного срабатывания Wake Word (РЕШЕНО):**
   - **Симптом**: После первой активации устройства по голосу («Компьютер») и последующего ответа, устройство больше не реагировало на Wake Word до перезагрузки.
   - **Причина**: Обнаружена логическая ошибка в `/main/application.cc` в функции `HandleWakeWordDetectedEvent`.

6.  **Кастомизация OLED для Breadboard Mini (2026-03-22):**
    -   **Проблема**: На OLED-экранах SSD1306 часто инвертированы цвета по умолчанию. Черная тема LVGL отображалась как белый светящийся прямоугольник с темным текстом.
    -   **Решение (Аппаратная инверсия)**: В `xingzhi-cube-0.96oled-wifi.cc` при инициализации добавлена команда `esp_lcd_panel_invert_color(panel_, true)`. Это аппаратно делает фон выключенным (черным) без нагрузки на процессор.
    -   **Настройки LVGL (`oled_display.cc`)**:
        *   Текст принудительно установлен в белый (`lv_color_white()`).
        *   Задний фон установлен в черный (`lv_color_black()`, `bg_opa COVER`).
    -   **Иконка**: Ввиду малого экрана микрочип заменен на статичный нейтральный смайлик (`FONT_AWESOME_NEUTRAL`).

7.  **Безусловный Wi-Fi (2026-03-22):**
    -   **Изменение**: Из конструктора платы `xingzhi-cube-0.96oled-wifi.cc` удалена проверка `ssid_manager.GetSsidList().empty()`.
    -   **Эффект**: Идентификатор сети `home4` теперь безусловно инжектируется при каждом запуске. Это решает проблему, когда плата могла «забыть» сеть или уходить в дефолтный Config-Mode после сброса параметров.

8.  **Фиксы звука и ОТА (Freenove & Breadboard) (2026-03-23):**
    -   **Громкость динамика**: Выставлена на **100%** для обоих типов плат. В `es8311_audio_codec.cc` добавлен `nullptr` чек для `dev_` для защиты от краша при регулировке до запуска потока.
    -   **Циклическое OTA (РЕШЕНО)**: Исправлен парсер JSON на сервере (`.102`), теперь он корректно считывает версию из раздела `"application"` вместо `"firmware"`. Версия в коде поднята до **2.8.0** для разрыва цикла бесконечного обновления.
    -   **Чувствительность Микрофона**: Аналоговое усиление `input_gain_` повышено с 30 dB до **35 dB** (эксперимент с 40 dB вызывал хрипы и сбои Вейкнета). Это улучшило реакцию на WakeWord с дистанции.

## 🔧 Заметки по OTA (2026-03-24)
- **Обновление версий**: При сборке прошивки для **реального обновления** девайсов, **ОБЯЗАТЕЛЬНО** нужно инкрементировать версию `PROJECT_VER` в файле `/xiaozhi-esp32/CMakeLists.txt` (например, `2.8.0` -> `2.8.1`).
- **Почему**: Сервер (`ota_server.py`) теперь динамически считывает версию из загруженного `.version` файла. Обновление выдается только если `версия_на_сервере > версия_на_плате`. Если они одинаковы, апдейт пропускается. Это предотвращает циклическое обновление.
