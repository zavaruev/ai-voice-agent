# 🚀 Стратегия публикации: AI Voice Agent

На основе анализа репозиториев (`xiaozhi-esp32`, `nanobot`, `speaches`) и локальных изменений, сформирована следующая структура для выноса проекта в паблик.

---

## 🔍 Текущее состояние
1.  **Прошивка (`xiaozhi-esp32`)**:
    *   *Дженерик фиксы*: Исправление бага Wake Word (`application.cc`), стабильность Wi-Fi.
    *   *Кастомизация*: Новые платы (`xingzhi-cube-0.96oled-wifi`, `freenove-2.8`), кастомный OLED (`esp_lcd_panel_invert_color`).
2.  **Шлюз (`nanobot`)**:
    *   *Ядро интеграции*: Файл `voice_server.py` (WebSocket, Opus-PCM, VAD, Whisper, TTS, MCP Proxy).
    *   *Статус*: Все файлы сейчас лежат локально (`untracked`), по сути это готовый модуль канала для Nanobot.
3.  **Служба STT (`speaches`)**:
    *   Используется "as is", без изменений кодовой базы, только конфигурация.
4.  **Развертывание**:
    *   Связано через Docker-Compose, локальные переменные и скрипты (`build_all.sh`, `ota_server.py`).

---

## 🗺️ Варианты и Оценка

### 1. Только Серия PR (Pull Requests)
*   **Плюсы**: Код поддерживается авторами оригиналов, максимальный охват.
*   **Минусы**:
    *   Кастомная плата `xingzhi-cube` может быть не нужна апстриму.
    *   `nanobot` может не захотеть принимать тяжеловесные аудио-зависимости (`opuslib`, `audioop`) в основную кодовую базу.
    *   **Нет единой точки входа** для сборки "системы под ключ".

### 2. Только Форки (Forks)
*   **Плюсы**: Полный контроль над кодом, можно зафиксировать версии.
*   **Минусы**: Пользователям придется клонировать 3 разных репозитория, вручную стыковать конфиги. Сложный порог входа.

---

## 💡 Рекомендация: Гибридный Подход (Оптимальный)

Для создания "Вау-эффекта" и простоты использования другими людьми, лучшим решением является **сборка в Единый Оркестратор / Дистрибутив (Bundle)**.

### Шаг 1: Апстрим (PR) — Вклад в сообщество
*   **`xiaozhi-esp32`**: Создать PR на исправление бага Wake Word (`application.cc`) и фикс Modem-Sleep. Это дженерик баги, их примут с радостью.
*   **`nanobot`**: Если авторы развивают систему плагинов — предложить `voice_server.py` как официальный Channel-плагин.

### Шаг 2: Единый Репозиторий ("Оркестратор")
Создать новый репозиторий на GitHub (например, `alexander/ai-voice-assistant`). 
Он **НЕ** должен содержать весь код `xiaozhi` или `nanobot`, его роль — **клей и дистрибуция**:

1.  **`docker-compose.yml`**: Который поднимает всю связку:
    *   `speaches` (официальный образ).
    *   `nanobot` (собранный из вашего форка/ветки с `voice_server.py`).
    *   `ota-server` / `ota-remedy` (образ на базе python).
2.  **`README.md`**: Красивая документация (на базе ваших заметкок) с архитектурой (Диаграмма: ESP32 -> WebSocket -> Nanobot -> STT -> LLM).
3.  **Папка `firmware/`**:
    *   Инструкции по сборке.
    *   Скрипт `build_all.sh`.
    *   Готовые бинарники (`.bin`) для популярных плат (`Freenove`, `Breadboard`), чтобы пользователи могли просто зашить девайс без настройки `idf.py`.
4.  **`config_template.json`**: Шаблоны настроек для Nanobot и Home Assistant.

## 🛠️ План действий (Как это сделать)

1.  **Подготовка кода**:
    *   В `xiaozhi-esp32` закомитить изменения в локальную ветку (например `feature/xingzhi-cube-fixes`).
    *   В `nanobot` закомитить `voice_server.py` и правки в Dockerfile.
2.  **Создание Оркестратора**:
    *   Инициализировать новый пустой репозиторий.
    *   Скопировать туда `build_all.sh`, `ota_server.py`, `docker-compose.yml`.
    *   Написать `README.md` с кнопкой "1-Click Setup" (или близком к этому).
